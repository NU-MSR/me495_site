Final Project (Fall 2019)

Important Dates

  1. Proposal Due: November 14th, before class
  2. Final Demonstration: December 9th from 3-5pm.
  3. Final Submission: Friday 14th 12:01 am.

Overview

The theme of this year's project is Recreational Robotics.

Each team will develop a ROS package that enables the robot to participate in a recreational activity with or on behalf of one or more humans. The contest is open-ended but will be judged on creativity, difficulty, and reliability.

Team assignments are posted on Canvas.

Requirements

The Robot's Tasks

  1. Manipulation: The robot must physically interact with at least one object in the environment using the robot's arm and or grippers. The robot must be move under its own power.

  2. Sensing: The solution must use at least one sensor. The behavior of the robot must change in response to the sensor reading.

  3. Human Interaction: A human user should be able to do something to change what the robot does (without needing to program). This can be while the robot is running (e.g., moving a piece on a chessboard) or can be part of the setup phase (e.g., arranging puzzle pieces).

The Students' Tasks

  1. Project Proposal (Due Nov 14th)
    • Brief (1-3 sentence) description of the project.
    • List of any external equipment required that may need to be purchased.
      • You should ask me about this before submitting your proposal so you don't write a proposal around an unobtainable item.
    • High level overview of your approach to the project
      • Important nodes to write
      • Third party packages to use
      • Concepts that are currently unfamiliar and must be learned
      • Anything that needs to be built Some light 3D printing or laser cutting is permissible, but there is no time for significant mechanical design or fabrication.
    • Description of the proposed role for each team member
    • Total proposal length should be under 2 pages.
  2. Source Code
    • Each team must use git to develop the project.
    • The project can consist of one or more ROS packages. If you use multiple packages in different git repositories or need to compile 3rd party packages in your workspace, provide a rosinstall file. Given the size of the project it most likely makes sense to have one git repository (although it may contain multiple packages)
    • The project must have multiple custom nodes, launch files, topics, and services.
    • Each student should work in their own branches in git. Working code should be merged into the master/ branch.
    • Follow the branch naming convention yourname/branch
    • You should have automated tests using rostest and unittest fo your code
  3. Documentation
    • All source code must be documented, with a docstring at the top describing the ROS API and docstrings for each function and class
    • There should be a single README.md describing the project, which can link to other documents as needed
      • Documentation can be done as a README.md that renders nicely on github or using github pages.
    • The README.md must include:
      1. Brief project overview
      2. Quickstart guide providing the exact commands
      3. List of all nodes and launchfiles and what they do
      4. A discussion of the overall system architecture and high level concepts
      5. Any other information, as appropriate (for example: algorithms used, lessons learned, future work)
      • Setup Sphinx to autogenerate your html API documentation
        • You may include more detailed documentation in sphinx rather than README.md and simply have The README.md link to sphinx documentation hosted on github.io for example, if you wish.
  4. Videos
    • You should produce a video showing your demonstration working.
    • The video should be hosted on line and linked to in your README or embedded in your project website.
  5. Live Demonstration
    • There will be a 15 minute oral presentation and demonstration on December 9th. No slides are required.

Grading

  1. The proposal
  2. Group Grade: All students receive the same grade based on:
    • How well the robot worked (weighed against the complexity of the task)
    • The documentation.
    • The quality of the python code and how well your package conforms to ROS standards.
    • How well git was used
  3. Grade adjustments:
    • I will get confidential feedback from each of you about your group members effort grades.
    • I will also look at the git history to see how much each individual contributed.
    • Grades may be adjusted up or down based on how your effort is perceived by your peers and demonstrating in the git history.

Potentially useful ROS Packages and Libraries

Tags

  • April Tags: Used for 3D pose estimation of a tag in space
  • AR Tags: An older form of 3D pose estimation with tags
  • Visual Servoing Uses vision to determine location of objects in space

Perception

  • OpenCV: Computer vision library, accessible through ROS
    • See opencvbridge For ROS support
  • PCL Point cloud library: see perceptionpcl for ROS support

Motion

  • MoveIt Motion planning library: sophisticated motion planning, beyond the built-in baxter/sawyer routines; however, it is more complicated to set up.
  • Joint Trajectory Action Server
    • Enables executing complex trajectories using software built-in to Baxter/Sawyer

Simulation

  • Baxter and Sawyer both have Gazebo support. See Rethink Robots for details.

State Machines

  • SMACH Provides a method for creating higher level state machines. This is a useful way to organize/switch between the higher level tasks in your robot.
