Part II: Turtle Control

Create a package that lets the simulated turtle follow a series of waypoints. The package you create depends on turtlesim because it is designed to control the turtle and uses messages/services from the turtlesim package.

  1. Create a package.xml and CMakeLists.txt in your turtle_control repository.
    • I recommend using catkin_create_pkg so you can read the comments in the generated files and learn about the format.
  2. Edit the package.xml to change the default but required fields to reflect your name, email, and a description of this package.
  3. Commit the changes and push to the remote.

Velocity Translator

Message: TurtleVel

Write a node called turtle_interpret that translates a custom velocity message into a geometry_msgs/Twist that can be used by the turtlesim. The node should be placed in your package's nodes directory. Be sure to make your python script executable (chmod 755).

  1. Create a new message (in the msg directory) called TurtleVel. The message should have the following fields:
    • linear: the linear velocity of the turtle in m/s
    • angular: the angular velocity of the turtle in rad/s.
    • Be sure to add comments (the # character begins a comment) describing the fields.
    • To properly add the message you will need to read CMakeLists.txt and adjust it and package.xml. There is a long comment block in CMakeLists.txt describing what you need to do.
    • The process of adding a service is nearly identical to that of adding a message.
  2. Make sure your package compiles cleanly with catkin_make and has no linting errors from catkin_lint before proceeding.
    • The errors from catkin_lint help explain why catkin_make fails.

Node: turtle_interpret

  1. Write a node called turtle_interpret that
    1. Subscribes to a topic turtle_vel, listening for TurtleVel messages.
      • The subscriber should log the input as a debug level message
      • The subscriber should publishes a corresponding geometry_msgs/Twist on cmd_vel.
    2. Offers a service called vel_translate that converts a geometry_msgs/Twist into a TurtleVel message and returns it.
      • Create a service file called VelTranslate.srv to define this service.
      • Your .srv will depend on the geometry_msgs package. You now must depend on this package in your package.xml.
      • The service should return None (indicating an error) if the twist has non-zero fields that don't correspond to the linear and angular velocity of a TurtleVel
      • Your project should compile with catkin_make and have no catkin_lint errors before proceeding.
    3. Use relative ROS names for everything so that it can be remapped and namespaced more easily
  2. Test the subscriber/publisher part of the node by using ROS commands to send messages while watching the ROS log and the cmd_vel topic.
    • To see the debugging message you will need to adjust the logger level of your node's rosout to debug.
    • You can use either rqt_logger_level or change settings in rqt_console.
  3. Test the service by using ROS commands to convert a geometry_msgs/Twist into a TurtleVel.

Turtle Waypoint

Write a node called waypoint that causes turtle1 to follow a series of waypoints (stored as a list of (x,y) positions). When the turtle reaches the last waypoint it should head to the first waypoint and restart the cycle.

Parameters

  1. The waypoints should be stored in a config/waypoint.yaml file which gets loaded into the Parameter Server and read by the node.
  2. There should be a private parameter called dist_thresh, which determines when the turtle is close enough to its destination to move to the next waypoint.

Subscribers

  1. The turtle may subscribe to the pose topic to read the pose, as reported by the turtle.

Publishers

  1. The node should publish a TurtleVel message on the turtle_vel topic to control the turtle.
    • Your turtle_interpret node will subscribe to turtle_vel and publish a cmd_vel for the turtle to follow

Startup

  • When the node is initialized, it should use the /spawn service spawn a new turtle called mark.
  • Prior to the node starting, mark should be moved to each waypoint location and run a small trajectory to draw an X marking each waypoint.
  • Then mark should lift its pen and teleport to the next waypoint (one waypoint ahead of turtle1)
    • Essentially, mark will always sit one waypoint ahead of turtle1
  • When turtle1 gets close to mark (based on dist_thresh), teleport mark to the next waypoint and have turtle1 head to it
  • If the final waypoint is reached, head back to the first waypoint and continue the cycle

Demonstration

  1. Write a launchfile called launch/waypoint_follow.launch that loads all the parameters, launches the turtlesim, turtle_interpret and waypoint nodes, and connects everything together to have the turtle follow the waypoints.

  2. Install simplescreenrecorder (via apt) or different screen recording software and use it to create a video of your code working.

  3. Upload your video to a sharing website, and include a link in your README.md

    • Alternatively, you can include your on Google Drive or Northwestern Box and link to it in your README.md (make sure I have access to it)
    • You can also convert the video to a gif and embed it in your README.md https://engineering.giphy.com/how-to-make-gifs-with-ffmpeg/

  4. Use a ROS command line tool to capture enough cmd_vel commands for the turtle to visit each waypoint twice.

    • One way to do this is to start roscore manually, then start the recording, then run rosbag. This ensures that you capture all the data from the beginning.
    • Add the bagged data as a waypoint.bag file to your repository.

  5. Kill the turtle_interpret node to stop the turtle from moving.

  6. Reset the turtle to its initial configuration using a turtlesim service.

  7. Use a ROS command line tool to play back your bagged data and make the turtle move.

    • You should not expect the trajectory to be duplicated, due to the inexact timing with ROS bags (see here)
