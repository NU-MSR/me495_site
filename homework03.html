Logistics

  1. Create a git repository using the link provided in Canvas.
  2. Your homework will be submitted via this git repository.
  3. I will grade whatever is on the master branch when I clone the repository.
  4. Guidelines provides important information about what you should turn-in, including information about your README.md file.
  5. All work on both parts of this problem should be done in a single git repository containing a single ROS package called armit
  6. In general, you should refer to ROS entities using relative names from within your nodes (this advice has been added to the Guidelines).

Robots

  1. This assignment will use the pincherX 100 robot.
    • Unlike the turtlebot3, all pincherX software runs on your computer
    • Follow these instructions to install the necessary packages, compile them, and ensure they are always on your path.
    • These are the same instructions as last time; however, you should delete and recreate the nuws to make sure you have the latest packages: many bugs have been fixed since then!
  2. Remote students who do not have access to the robots will complete the project in simulation, however, the software setup is the same for all students.

Part I: Gazebo World

  1. Create a gazebo world as follows:
    • There is a tall rectangular obstacles directly behind the robot. This obstacle is fixed
      • You can use whatever obstacle you want, I used the bookshelf.
    • There is a rectangle representing the Intel Realsense D435i Box. It is 14cm x 9cm x 5 cm and weight 100g
      • Place the RealSense box directly in front of the robot, within reach
  2. Running roslaunch homework3 arm_box.launch should start the gazebo world, and spawn the robot
  3. Store the world in the worlds directory of your package
  4. Store the model for the box in the models directory for the package
  5. Write a node called batter that in response to a service called swing commands the robot arm to smack the box as hard as possible.
    • Only use this service in gazebo not with the real robot
  6. Write a launchfile called goodbye.launch that launches gazebo and runs the whack node

Hints

  1. Copy the launchfile used by the px100 gazebo program to arm_box.launch and modify it to meet the specifications
  2. You may use the gazebo world editor to create the world file and the model of the box. Then use the modified px100 launchfile to load your world
  3. The default interbotix gazebo launchfile starts gazebo in a paused state.
    • You can call a service to un-pause gazebo or press the play button in the gazebo window
  4. Note that all parameters/names used by the interbotix arm are stored under the /px100 namespace. This means that you will need to specify/remap names throughout this assignment so that they use the appropriate values in the /px100 namespace (or launch your nodes in the px100 namespace).

Part II: Motion Planning

  1. You can choose to do this final task either on the robot or in gazebo.
  2. Place a RealSense Box in front of your arm (in the real world or gazebo)
  3. Place a collision object to represent the table
  4. Create a node called grabber that implements the following:
    1. A reset service that
      • Takes an (x,y) position of the realsense box and creates/moves a planning scene box at that location
        • The box can be oriented in a way that makes it easier for the arm to pick it up, such as always aligning it radially from the arm
      • Moves the robot to its pre-defined 'Home' position without hitting the box ('Home' is pre-defined by the SDK)
      • If you are doing your project in gazebo, this service should move the box to the correct position in gazebo
    2. A pickup service.
      • Use moveit motion planning commands to pick up the box and move it to a specified (x,y) position on the table
      • When you grab the box, you should attach the box to the arm
      • When you release the box, you should de-attach from the arm and add to the planning scene
      • Return to the home position when done.
  5. Write a launchfile called arm.launch
    • Launch the moveit interface by including the launchfile from interbotix_ros_arms using an <include> tag
    • You may also look to that launchfile for inspiration and customize it if you would like (for example to load a custom rviz configuration)
    • Everyone's launchfile should take an argument to let you launch the arm with use_fake.
      • Then you should also launch in gazebo if using gazebo or on the real robot if using the real robot

Hints

  1. It is often easier to launch gazebo and keep it running. Then just kill and manually restart your node
  2. You may wish to do the bulk of your testing with the fake MoveIt node.
  3. You may wish to run through some tutorials prior to attempting the assignment: -MoveGroup Python Tutorial
    • Interbotix MoveIt Tutorial
    • Everything you need to do with MoveIt to complete this assignment is contained in the aforementioned tutorial
    • I do not recommend using MoveIt's grasping functionality for this assignment.
  4. When you launch the interbotix moveit gui in rviz, you can use moveit interactively and look up lots of information about the robot such as
    • It's Planning groups
    • It's built in positions
  5. At first, just try to make the robot do something simple in code.
    • A good first task is to make it toggle between the Home and Upright states.
    • API reference is here
      • set_named_target() may help
    • A good second step is to add the box and the wall to the planning scene and make sure they show up in rviz
  6. At some points you may care about position, and others orientation. You can set_position_target to set position without respect to pose and set_orientation_target or set_rpy_target to set one without regard for the other.
    • Remember, this robot has fewer than 6-dof, so many 6-DOF poses of the end effector are invalid.
  7. shift_pose_target() is also helpful
  8. You can try out most things in the RViz GUI prior to coding them
  9. There are two tricky parts to PlanningSceneInterface that you must work around
    1. After creating the PlanningSceneInterface, it's publishers may not have necessarily finished connecting to their respective topics
      • This means that you can potentially add an object prior to the PlanningSceneInterface being ready, in which case the object will not get added
    2. Your node could exit after modifying the planning scene but before the message is published.
    3. To solve these issues, you should, after every action on a planning scene, wait (with a timeout) to see if your changes actually took effect
      • Sample code for doing this is provided in the tutorial
      • Depending on the context, the action you take after a timeout failure is different
      • For example, if you just started the planning scene, you may wish to retry the addition a few times upon timing out
